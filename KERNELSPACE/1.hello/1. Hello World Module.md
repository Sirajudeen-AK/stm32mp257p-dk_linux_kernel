# Program 1 — Hello World Module

## 1. Goal

Learn the absolute basics of a Linux Kernel Module (LKM):

- How a module enters the kernel (`module_init`)
- How a module leaves the kernel (`module_exit`)
- How to load, unload, and view kernel logs (`insmod`, `rmmod`, `dmesg`)

A kernel module is a piece of code that can be loaded into and unloaded from
the running kernel **without rebooting**. Think of it as a "plugin" for the kernel.

---

## 2. How It Works (Flow)

```
User space                 Kernel space
----------                 ------------
insmod hello.ko  ───────▶  init_module() syscall
                              │
                              ▼
                       my_init()  (your module_init function)
                              │
                              ▼
                       printk()  ───────▶  kernel log buffer (read via dmesg)

rmmod hello      ───────▶  delete_module() syscall
                              │
                              ▼
                       my_exit()  (your module_exit function)
```

---

## 3. Code Walkthrough

```c
static int __init my_init(void)   // runs once when module loads
{
    printk(KERN_INFO "Hello, Kernel!\n");
    return 0;                     // 0 = success, negative = load failed
}

static void __exit my_exit(void)  // runs once when module unloads
{
    printk(KERN_INFO "Goodbye, Kernel!\n");
}

module_init(my_init);   // register entry point
module_exit(my_exit);   // register exit point

MODULE_LICENSE("GPL"); // avoids "kernel tainted" warning
```

- `__init` — marks the function so its memory is **freed after init finishes**.
- `__exit` — marks cleanup code; dropped entirely if the module is built-in.

---

## 4. Commands

```bash
make                 # build hello.ko
sudo insmod hello.ko # load the module
lsmod | grep hello   # confirm it is loaded
sudo rmmod hello     # unload the module
dmesg | tail         # see the printk messages
modinfo hello.ko     # view module metadata (license, vermagic, etc.)
```

## 5. Interview Questions & Answers

**Q1. What is the difference between a built-in driver and a loadable module?**

> A built-in driver is compiled directly into the kernel image (`vmlinux`) and is
> always present. A loadable module (`.ko`) is compiled separately and can be
> inserted/removed at runtime with `insmod`/`rmmod`, saving memory until needed.

**Q2. What is `vermagic`?**

> A version-magic string stored inside the `.ko` (kernel version, compiler, SMP,
> etc.). The kernel refuses to load a module whose `vermagic` does not match the
> running kernel, preventing crashes from ABI mismatches.

**Q3. Why use `__init`?**

> The init function is only needed once at load time. Marking it `__init` lets the
> kernel free that memory after initialization completes, reducing RAM usage.

**Q4. Why use `__exit`?**

> `__exit` marks cleanup code that runs during `rmmod`. If the module is compiled
> built-in (cannot be removed), the kernel discards this code entirely.

**Q5. Why does `module_init` return an `int`?**

> Returning `0` means success. Returning a negative errno (e.g. `-ENOMEM`) tells
> the kernel that loading failed, so the module is rejected and not inserted.

**Q6. Where does `printk` output go and why not `printf`?**

> `printk` writes to the kernel ring buffer, viewed with `dmesg`. `printf` is a
> user-space C library function and does not exist in kernel space.

**Q7. What does `MODULE_LICENSE("GPL")` do?**

> It declares the module's license. Without a GPL-compatible license the kernel
> marks itself "tainted" and hides GPL-only symbols from the module.
