## Interview Questions & Answers

**Q1. Why use request_threaded_irq() instead of request_irq()?**

> Because I²C and Regmap operations can sleep. Threaded IRQ handlers execute in process
> context, where sleeping is permitted.

**Q2. Why is the primary (top-half) handler NULL?**

> No immediate hard IRQ work is required. Linux masks the interrupt and directly schedules
> the threaded handler.

**Q3. Why read INT_STATUS first?**

> Reading INT_STATUS acknowledges the interrupt source and verifies that the interrupt was
> generated for the expected event before processing data.

**Q4. Why use regmap_bulk_read() inside the handler?**

> The accelerometer outputs are stored in consecutive registers, so one bulk transaction
> is faster and keeps the X/Y/Z sample coherent.

**Q5. Why return IRQ_NONE?**

> If the interrupt wasn't generated by this device (for example, INT_STATUS doesn't
> indicate a data-ready event), returning IRQ_NONE tells the kernel this handler did not
> service the interrupt.

**Q6. Where do you clear the interrupt in your driver?**

> We don't explicitly clear it. The MPU-9250's data-ready interrupt is acknowledged
> by reading the interrupt status register and/or the sensor data registers, depending
> on the INT_PIN_CFG configuration. There is no dedicated interrupt-clear register.

> **Note:** Use `IRQF_TRIGGER_FALLING` only when it matches the DTS (`IRQ_TYPE_EDGE_FALLING`).
> In many modern Device Tree drivers the trigger type is already described in the DTS so
> the driver often only passes `IRQF_ONESHOT`. Keeping them consistent is fine for
> learning — but never let them disagree.
