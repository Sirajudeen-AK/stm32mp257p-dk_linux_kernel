# Program 6 — ioctl (Device Control Interface)

## 1. Goal

Learn how to add a **control/configuration channel** to a character driver using
`ioctl` (I/O control). `read`/`write` are for moving data; `ioctl` is for
*commands* like "set baud rate", "reset device", "get status".

---

## 2. How It Works

### Defining commands (in a shared header)

```c
#define MY_MAGIC 'k'
#define MY_SET_VALUE _IOW(MY_MAGIC, 1, int)  // user -> kernel
#define MY_GET_VALUE _IOR(MY_MAGIC, 2, int)  // kernel -> user
```

- `_IOW` = command that **writes** data to the kernel.
- `_IOR` = command that **reads** data from the kernel.
- `_IOWR` = does both.

### Driver side

```c
static long my_ioctl(struct file *f, unsigned int cmd, unsigned long arg)
{
    int value;
    switch (cmd) {
    case MY_SET_VALUE:
        if (copy_from_user(&value, (int __user *)arg, sizeof(value)))
            return -EFAULT;
        break;
    case MY_GET_VALUE:
        value = 42;
        if (copy_to_user((int __user *)arg, &value, sizeof(value)))
            return -EFAULT;
        break;
    default:
        return -ENOTTY;   // unknown command
    }
    return 0;
}

static struct file_operations fops = {
    .owner          = THIS_MODULE,
    .unlocked_ioctl = my_ioctl,
};
```

### User side

```c
int v = 100;
ioctl(fd, MY_SET_VALUE, &v);
ioctl(fd, MY_GET_VALUE, &v);
```

---

## 3. Flow

```
User: ioctl(fd, MY_SET_VALUE, &v)
          │
          ▼
VFS routes by command number
          │
          ▼
Driver: my_ioctl()  → switch(cmd) → copy_from_user/copy_to_user
```

---

## 4. Interview Questions & Answers

**Q1. Why use `ioctl` instead of `read`/`write`?**

> `read`/`write` are designed to transfer a **stream of data**. `ioctl` is for
> **control and configuration** operations that don't fit a simple byte stream —
> e.g. setting modes, querying status, or issuing device-specific commands.

**Q2. Why call `copy_from_user`/`copy_to_user` inside `ioctl`?**

> The `arg` parameter is usually a **user-space pointer**. The kernel cannot
> dereference user memory directly, so it must use the copy helpers to move the
> data safely across the user/kernel boundary.

**Q3. Why `.unlocked_ioctl` and not `.ioctl`?**

> Old kernels used `.ioctl`, which was called while holding the Big Kernel Lock
> (BKL). The BKL was removed, so modern kernels use `.unlocked_ioctl`, letting the
> driver do its own fine-grained locking instead of relying on a global lock.

**Q4. What is `arg` in `long my_ioctl(..., unsigned long arg)`?**

> It is the third argument passed by the user. It can be an integer value, or —
> most commonly — a **pointer** (to an int or a struct) in user space, which the
> driver accesses via `copy_from_user`/`copy_to_user`.

**Q5. Why use the `_IO`/`_IOR`/`_IOW` macros instead of plain numbers?**

> They encode a magic number, command number, direction, and data size into the
> command code. This catches mismatched commands (wrong device or wrong struct)
> and makes the interface self-describing and safer.

**Q6. What should the `default` case return for an unknown command?**

> `-ENOTTY` ("Inappropriate ioctl for device"). It is the conventional errno that
> tells user space the command is not supported by this driver.
