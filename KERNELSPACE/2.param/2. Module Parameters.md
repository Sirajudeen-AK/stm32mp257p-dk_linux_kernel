# Program 2 — Module Parameters

## 1. Goal

Learn how to pass values into a module **at load time** (and even change them at
runtime) using `module_param()`. This lets one driver behave differently without
recompiling — for example, setting a debug level or a buffer size.

---

## 2. How It Works

```c
static int number = 10;                 // default value
module_param(number, int, 0644);        // name, type, sysfs permission
MODULE_PARM_DESC(number, "An example integer parameter");
```

When you load the module you can override the default:

```bash
sudo insmod param.ko number=123
```

Because the permission is `0644`, the value is also exposed in sysfs and can be
read (and written, by root) at runtime:

```
/sys/module/<module_name>/parameters/number
```

```bash
cat /sys/module/param/parameters/number      # read current value
echo 999 > /sys/module/param/parameters/number # change it live (root)
```

### Flow

```
insmod param.ko number=123
        │
        ▼
kernel parses "number=123"
        │
        ▼
writes 123 into the variable `number`   ← happens BEFORE module_init()
        │
        ▼
module_init() runs and sees number = 123
        │
        ▼
sysfs file created:
/sys/module/param/parameters/number
        │
        ▼
cat  ──► read the live value
echo ──► change the live value (root, if writable)
```

## 3. Common Parameter Types

| Type macro              | C type        |
| ----------------------- | ------------- |
| `module_param(x, int, ...)`   | `int`   |
| `module_param(x, charp, ...)` | `char *` (string) |
| `module_param(x, bool, ...)`  | `bool`  |
| `module_param_array(arr, int, &count, ...)` | array of int |

---

## 4. Permission Bits (the 3rd argument)

| Value  | Meaning                                   |
| ------ | ----------------------------------------- |
| `0`    | Not visible in sysfs at all               |
| `0444` | Read-only for everyone                    |
| `0644` | Read for all, write for root              |
| `0666` | Read/write for all (avoid — insecure)     |

## 5. Interview Questions & Answers

**Q1. How are parameters passed to a module during `insmod`?**

> As `name=value` pairs on the command line, e.g. `insmod param.ko number=123`.
> The kernel parses them and writes them into the variables registered with
> `module_param()` **before** `module_init()` runs.

**Q2. What is the difference between a module parameter and a sysfs attribute?**

> A module parameter is the simplest way to expose a single variable; the kernel
> auto-creates the sysfs file for you under `/sys/module/.../parameters/`.
> A custom sysfs attribute requires you to write `show`/`store` callbacks but
> gives full control (validation, formatting, side effects).

**Q3. Why is permission `0644` commonly used?**

> It lets any user read the current value while restricting writes to root.
> `0644` = owner read/write, group read, others read.

**Q4. Can a module parameter be changed after the module is loaded?**

> Only if it was given a writable sysfs permission (e.g. `0644`). Then writing to
> `/sys/module/<mod>/parameters/<name>` updates the live variable. With permission
> `0`, it can only be set at load time.

**Q5. What is `MODULE_PARM_DESC` for?**

> It attaches a human-readable description to the parameter, shown by
> `modinfo <module>.ko`. Purely documentation — it does not affect behavior.
