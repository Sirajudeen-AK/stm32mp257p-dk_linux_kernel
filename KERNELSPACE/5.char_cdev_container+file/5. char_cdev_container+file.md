# Program 5 вЂ” file->private_data and container_of()

## 1. Goal

Give **each open file (and each device instance) its own private state**, instead
of sharing global variables. Two key tools make this possible:

- `file->private_data` вЂ” a per-open pointer the driver can use freely.
- `container_of()` вЂ” recovers the enclosing struct from a member pointer.

---

## 2. The Problem with Globals

```c
char global_buffer[128];   // вќЊ every open() shares this same buffer
```

If two processes open the device, they overwrite each other's data because the
state is global. We want **separate state per device**.

---

## 3. The Solution

### Step 1 вЂ” Embed `cdev` inside your own device struct

```c
struct mydev {
    char buffer[128];
    wait_queue_head_t wq;
    struct cdev cdev;      // embedded, not a pointer
};
```

### Step 2 вЂ” In `open()`, recover the device with `container_of()`

```c
static int my_open(struct inode *inode, struct file *file)
{
    struct mydev *dev = container_of(inode->i_cdev, struct mydev, cdev);
    file->private_data = dev;     // stash it for later calls
    return 0;
}
```

### Step 3 вЂ” In `read()/write()`, read it back

```c
static ssize_t my_read(struct file *file, char __user *buf, size_t n, loff_t *off)
{
    struct mydev *dev = file->private_data;   // no globals needed
    // ... use dev->buffer, dev->wq ...
}
```

---

## 4. How container_of() Works

`container_of(ptr, type, member)` does pointer arithmetic: given a pointer to a
**member**, it subtracts the member's offset to get a pointer to the **whole
struct**.

```
inode->i_cdev  в”Ђв”Ђв”Ђв”Ђв–¶  &dev->cdev
                          в”‚  container_of() subtracts offsetof(struct mydev, cdev)
                          в–ј
                       struct mydev *   (the full object)
```

**Benefit:** multiple minors can share one driver while each keeps its own
buffer, wait queue, and state.

## 6. Interview Questions & Answers

**Q1. What is an `inode` vs a `file`?**

> An `inode` represents the **device/file itself on disk or in the VFS** вЂ” there is
> one per device node, shared by all who open it. A `struct file` represents **one
> open instance** вЂ” a new one is created on every `open()`, holding the offset,
> flags, and `private_data` for that specific open.

**Q2. What is the difference between `inode` and `file`?**

> One `inode` can have **many** `struct file` objects pointing to it (one per
> open). The inode is the static identity of the device; the file is the dynamic,
> per-open session state.

**Q3. Why not just use global variables?**

> Globals are shared by every open and every device instance, so concurrent users
> corrupt each other's data. Per-open/per-device state avoids this and lets one
> driver serve many devices safely.

**Q4. Why use `file->private_data`?**

> It is a convenient per-open pointer slot. Setting it in `open()` lets every
> later `read`/`write`/`ioctl` instantly find its device object without searching
> a global table.

**Q5. How are separate device instances maintained?**

> Each device gets its own `struct mydev` (with embedded `cdev`). In `open()`,
> `container_of(inode->i_cdev, ...)` finds the correct instance and stores it in
> `file->private_data`, so each instance keeps its own buffer and state.

**Q6. Why is `cdev` embedded inside the struct rather than a pointer?**

> Embedding it means `container_of()` can compute the address of the enclosing
> struct from the `cdev` address. With a pointer you'd lose that relationship and
> need a separate lookup.
