# Program 9 — wake_up_interruptible_nr()

## 1. Goal

Wake **only a limited number** of waiting processes instead of all of them, to
avoid the **thundering herd problem** introduced in Program 8.

```c
wake_up_interruptible(&wq);       // wakes ALL waiters
wake_up_interruptible_nr(&wq, 1); // wakes AT MOST 1 waiter
```

**Experiment:** run several user-space `poll()` apps, write data once, and observe
that only one waiter consumes the event.

---

## 2. The Problem (Thundering Herd)

Several apps all sleep on the same wait queue:

```
app1 --> poll() \
app2 --> poll()  \
app3 --> poll()   >-- all sleeping on  wait_queue_head_t my_wq
app4 --> poll()  /
```

A writer triggers a wake-up:

```bash
echo hello > /dev/mychar      # driver calls wake_up_interruptible(&my_wq)
```

Result — **all four wake**, but only one gets the single piece of data; the other
three wake up for nothing and go back to sleep. This wasted churn is the
**thundering herd problem**.

---

## 3. The Solution

```c
wake_up_interruptible_nr(&my_wq, 1);   // wake at most 1 sleeper
```

```
| Function                          | Wakes              |
| --------------------------------- | ------------------ |
| wake_up_interruptible(&wq)        | ALL sleeping waiters |
| wake_up_interruptible_nr(&wq, N)  | At most N sleeping waiters |
```

---

## 4. Important Caveat

Neither function **guarantees** that only N processes will ever *observe* the
event. Other processes may later call `poll()` or `read()` while the condition is
still true and see it too. `_nr` only limits how many **currently sleeping**
waiters are woken in that single call — it does not gate future observers.

If you need strict one-waiter-at-a-time wake-up, use exclusive waits
(`wait_event_interruptible_exclusive` + `wake_up_interruptible`), where the
kernel wakes exclusive waiters one at a time.

---

## 5. Interview Questions & Answers

**Q1. What is the thundering herd problem?**

> When many processes sleep on the same event and a single wake-up wakes them all,
> but only one can make progress. The rest wake, find no work, and sleep again —
> wasting CPU and causing scheduler/lock contention.

**Q2. Why is `wake_up_interruptible_nr()` better than `wake_up_interruptible()`?**

> It limits how many sleeping waiters are woken (e.g. just one), so you don't
> needlessly wake processes that cannot consume the data. This reduces wasted
> wake-ups when only a limited amount of work is available.

**Q3. Does `wake_up_interruptible_nr(&wq, 1)` guarantee only one process ever sees the event?**

> No. It only wakes at most one **currently sleeping** waiter. Other processes can
> still observe the condition later via a fresh `poll()`/`read()` while it remains
> true. For strict serialization use exclusive waits.

**Q4. What is the difference between `_nr` and exclusive wait queues?**

> `_nr` caps the number woken in one call but still wakes non-exclusive waiters
> from the front. Exclusive waiters (added with `_exclusive`) are woken **one per
> wake-up call** and placed at the tail, which is the cleaner way to hand off work
> to exactly one consumer at a time.
