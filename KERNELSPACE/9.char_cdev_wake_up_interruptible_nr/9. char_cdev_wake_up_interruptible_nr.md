# Program 9 - wake_up_interruptible_nr()

Goal: Avoid waking every waiter (thundering herd problem).

Key idea:
wake_up_interruptible(&wq);     -> wakes all waiters
wake_up_interruptible_nr(&wq,1); -> wakes only one waiter

Expected experiment:
Run multiple user-space poll() applications.
Write data once.
Observe only one waiter consuming the event.


## Problem

10 processes waiting.

wake_up_interruptible()

All 10 wake.
Only 1 gets data.

This is thundering herd.

## Solution

wake_up_interruptible_nr(&wq,1);

Only one waiter wakes.

## Interview Questions
Why better than wake_up_interruptible?
What is thundering herd problem?


Problem
Suppose:

app1 --> poll()
app2 --> poll()
app3 --> poll()
app4 --> poll()

All are sleeping on same waitqueue:

wait_queue_head_t my_wq;

Writer:
echo hello > /dev/mychar

calls:
wake_up_interruptible(&my_wq);

Result:

app1 wake
app2 wake
app3 wake
app4 wake

All wake.

Only one gets data.

Others wake unnecessarily.

This is called:
Thundering Herd Problem



Difference between:

wake_up_interruptible()
and
wake_up_interruptible_nr(&wq,1)

is:

wake_up_interruptible():
    Wake all sleeping waiters.

wake_up_interruptible_nr():
    Wake at most N sleeping waiters.

But neither guarantees:

Only N processes will observe the event.

Because other processes may later execute poll() or read() while the condition remains true.
