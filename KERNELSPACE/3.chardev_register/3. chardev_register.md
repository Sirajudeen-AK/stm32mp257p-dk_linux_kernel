# Program 3 — register_chrdev()

## 1. Goal

Register a **character device** with the kernel using the classic
`register_chrdev()` API, and understand how a user-space call like `open()` or
`read()` on `/dev/xxx` ends up calling **your** driver functions.

---

## 2. Major and Minor Numbers

Every device node has two numbers:

- **Major number → identifies the driver** that handles the device.
- **Minor number → identifies a specific device instance** managed by that driver.

```
/dev/null   => major 1  minor 3
/dev/i2c-0  => major 89 minor 0
```

You can see them with `ls -l /dev` (the two numbers before the date).

---

## 3. VFS Flow (how user calls reach your driver)

```
User space            Kernel (VFS)               Your driver
----------            -------------              -----------
open("/dev/x")  ───▶  VFS looks up major  ───▶  file_operations.open()
read(fd, ...)   ───▶  VFS routes by fd    ───▶  file_operations.read()
write(fd, ...)  ───▶  VFS routes by fd    ───▶  file_operations.write()
close(fd)       ───▶  VFS                 ───▶  file_operations.release()
```

The glue is the **`struct file_operations`** table — it maps each syscall to one
of your functions:

```c
static struct file_operations fops = {
    .owner   = THIS_MODULE,
    .open    = my_open,
    .release = my_release,
    .read    = my_read,
    .write   = my_write,
};

major = register_chrdev(0, "mychardev", &fops); // 0 = let kernel pick major
```

---

## 4. Commands

```bash
sudo insmod chardev.ko
dmesg | tail                       # find the assigned major number
sudo mknod /dev/mychardev c <major> 0   # create the device node
echo hi > /dev/mychardev           # triggers my_write()
cat /dev/mychardev                 # triggers my_read()
sudo rmmod chardev
```

## 5. Interview Questions & Answers

**Q1. Why do we need `struct file_operations`?**

> It is the dispatch table that connects user-space syscalls (`open`, `read`,
> `write`, `ioctl`...) to the driver's implementation. The VFS uses it to know
> which function to call for each operation on the device.

**Q2. What are `open` and `release` used for?**

> `open` runs when a process opens the device — used to allocate resources or set
> up per-open state. `release` runs on the **last** `close()` of that file — used
> to free those resources. Note: `release` is not called on every `close`, only
> when the final reference is dropped.

**Q3. Difference between a character device and a block device?**

> A character device transfers data as a **stream of bytes**, accessed sequentially
> (e.g. serial port, keyboard). A block device transfers data in **fixed-size
> blocks** with random access and a buffer cache (e.g. disks, SD cards).

**Q4. What does passing `0` as the major to `register_chrdev()` do?**

> It asks the kernel to **dynamically allocate** a free major number, returned as
> the function's value. Passing a non-zero number requests that specific major.

**Q5. Why is `register_chrdev()` considered "old"?**

> It reserves a whole range of 256 minors and is less flexible. Modern drivers use
> `alloc_chrdev_region()` + `cdev_init()` + `cdev_add()` for fine-grained control
> over minor ranges and multiple devices (covered in later programs).
