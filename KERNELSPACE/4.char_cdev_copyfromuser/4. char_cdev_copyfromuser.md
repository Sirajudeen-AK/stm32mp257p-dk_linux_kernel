# Program 4 — copy_to_user / copy_from_user + cdev Framework

## 1. Goal

Two connected ideas:

1. **Safely move data between user space and kernel space** using
   `copy_to_user()` and `copy_from_user()`.
2. Register the character device the **modern** way with the `cdev` framework
   (`alloc_chrdev_region` + `cdev_init` + `cdev_add`).

---

## 2. Why copy_to_user / copy_from_user?

User space and kernel space use **separate, isolated address spaces**. A pointer
from user space is *not* directly valid in the kernel, so the kernel must never
dereference it directly.

```c
// ❌ WRONG — direct dereference of a user pointer (can crash the kernel)
kernel_buf[0] = *user_buf;

// ✅ CORRECT — validated, safe copy
copy_from_user(kernel_buf, user_buf, len);  // user  -> kernel
copy_to_user(user_buf, kernel_buf, len);    // kernel -> user
```

These helpers **validate** the user pointer and safely handle page faults. They
return the **number of bytes that could NOT be copied** (0 means full success).

```c
if (copy_from_user(kbuf, ubuf, len))
    return -EFAULT;   // some bytes failed -> report bad address
```

### Flow (the user/kernel boundary)

```
   USER SPACE                          KERNEL SPACE
   ----------                          ------------

   write(fd, ubuf, len)
        │
        ▼
   ubuf  ───────── copy_from_user() ──────►  kbuf
                   (checks + safe fault handling)

   read(fd, ubuf, len)
        ▲
        │
   ubuf  ◄──────── copy_to_user() ─────────  kbuf
```

The two address spaces never touch each other directly — the copy helpers are the
only safe bridge across the boundary.

## 3. The cdev Framework (modern registration)

```c
dev_t dev;
struct cdev my_cdev;

alloc_chrdev_region(&dev, 0, 1, "mychardev"); // get major+minor dynamically
cdev_init(&my_cdev, &fops);                   // bind file_operations
cdev_add(&my_cdev, dev, 1);                   // activate the device
```

### Registration Flow

```
alloc_chrdev_region()   ──►  reserves major + minor (dev_t)
        │
        ▼
cdev_init(&my_cdev, &fops)  ──►  links your file_operations
        │
        ▼
cdev_add(&my_cdev, dev, 1)  ──►  device goes LIVE in the kernel
        │
        ▼
   /dev node usable  ──►  open / read / write reach your driver

   (cleanup runs in reverse)
   cdev_del()  ──►  unregister_chrdev_region()
```

Cleanup (reverse order):

```c
cdev_del(&my_cdev);
unregister_chrdev_region(dev, 1);
```

## 4. Interview Questions & Answers

**Q1. What happens if the user passes an invalid pointer?**

> `copy_to_user`/`copy_from_user` detect it and return a non-zero count (the bytes
> not copied). The driver then returns `-EFAULT`. The kernel does **not** crash —
> that is exactly why these helpers exist.

**Q2. Why is directly dereferencing a user pointer dangerous?**

> The user address may be invalid, unmapped, swapped out, or maliciously crafted.
> Dereferencing it in kernel context can cause an oops/panic or a security hole.
> The copy helpers add access checks and safe fault handling.

**Q3. Difference between `EFAULT` and `EINVAL`?**

> `-EFAULT` = "Bad address" — a memory pointer was invalid (copy failed).
> `-EINVAL` = "Invalid argument" — the *value/argument* was wrong (e.g. a bad size
> or unsupported command), but the pointer itself was fine.

**Q4. What is the difference between `register_chrdev()` and the cdev framework?**

> `register_chrdev()` is the legacy one-call API that grabs 256 minors at once.
> The cdev framework (`alloc_chrdev_region` + `cdev_init` + `cdev_add`) gives
> precise control over the major/minor range and cleanly supports multiple
> devices — it is the recommended modern approach.

**Q5. Why is `struct cdev` often embedded inside the device structure?**

> Embedding `cdev` inside your own device struct lets you recover the full device
> object from the `cdev` pointer using `container_of()` (covered in Program 5).
> This keeps per-device state together and supports many device instances cleanly.

**Q6. Do `copy_to_user`/`copy_from_user` always copy everything?**

> Not guaranteed — they return the number of bytes **not** copied. Always check the
> return value; a non-zero result means a partial/failed copy and you should
> return `-EFAULT`.
