# Program 7 — Wait Queues (Sleeping & Waking a Process)

## 1. Goal

Learn how a driver can put a process **to sleep** until data is available, and
**wake it up** later — without burning CPU in a busy loop. This is the foundation
for blocking reads, `poll`, and event-driven drivers.

---

## 2. The Problem

If there is no data yet, a naive driver might spin:

```c
while (no_data)        // ❌ busy-wait: wastes 100% CPU
    ;
```

Instead, the process should **sleep** and let other work run, then be woken when
data arrives.

---

## 3. The Wait Queue Mechanism

```c
// 1. Declare a wait queue (often inside the device struct)
wait_queue_head_t my_wq;
init_waitqueue_head(&my_wq);

// 2. Reader sleeps until the condition becomes true
wait_event_interruptible(my_wq, data_ready != 0);

// 3. Writer / ISR makes data available and wakes sleepers
data_ready = 1;
wake_up_interruptible(&my_wq);
```

- `wait_event_interruptible(wq, cond)` — sleeps until `cond` is true; can be
  interrupted by a signal (returns `-ERESTARTSYS`).
- `wake_up_interruptible(&wq)` — wakes processes sleeping on `wq` so they
  re-check the condition.

---

## 4. Flow

```
User: read()
        │
        ▼
Driver: my_read()  → no data  → wait_event_interruptible()  (SLEEP)
                                          ▲
                                          │ wake_up_interruptible()
Writer / ISR  ────── data_ready = 1 ──────────┘
        │
        ▼
read() resumes  → copy_to_user()  → return
```

---

## 5. Why `wait_event_interruptible` checks a condition

A process can be woken "spuriously" or by an unrelated event, so the macro
**re-checks the condition** in a loop. It only truly returns when the condition is
actually true — this prevents acting on a false wake-up (a classic bug source).

---

## 6. Interview Questions & Answers

**Q1. What is a wait queue?**

> A kernel list of processes sleeping until a particular condition becomes true.
> A driver puts a process on the queue with `wait_event_*` and removes/wakes them
> with `wake_up_*`, enabling efficient blocking without polling.

**Q2. Difference between `wait_event` and `wait_event_interruptible`?**

> `wait_event` sleeps in an **uninterruptible** state — signals cannot wake it.
> `wait_event_interruptible` can be interrupted by a signal (e.g. Ctrl+C) and
> returns `-ERESTARTSYS`, which the driver should handle. The interruptible form
> is preferred so processes remain killable.

**Q3. Why not use a `while(1)` busy loop to wait for data?**

> Busy-waiting keeps the CPU 100% busy doing nothing useful, wastes power, and
> starves other tasks. Sleeping on a wait queue frees the CPU until data is ready.

**Q4. What wakes a sleeping reader?**

> A call to `wake_up_interruptible(&wq)` from the writer path or an ISR, made after
> the wake condition (e.g. `data_ready = 1`) has been set.

**Q5. Why must you set the condition flag *before* calling `wake_up`?**

> Because the woken process re-checks the condition. If the flag isn't set first,
> the process wakes, finds the condition false, and goes right back to sleep —
> losing the event.

**Q6. What does `wait_event_interruptible` return and why check it?**

> It returns `0` if the condition became true, or `-ERESTARTSYS` if a signal
> interrupted the sleep. Checking it lets the driver abort cleanly instead of
> proceeding as if data had arrived.
