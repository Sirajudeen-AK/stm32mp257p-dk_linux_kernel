# Program 8 — poll / select / epoll

## 1. Goal

Allow a user program to **wait on one or many file descriptors at once** and be
notified when any of them becomes readable/writable — without busy-waiting and
without blocking on a single `read()`. This is how event-driven apps (servers,
GUIs) scale.

---

## 2. The Problem

```c
while (1)        // ❌ busy loop: wastes CPU and handles only one fd well
    read(fd, buf, n);
```

We want the kernel to tell us *when* data is ready, across many fds.

---

## 3. The Solution

User space picks one of these (all do the same core job, with different scaling):

| API      | Scales to many fds | Notes                                  |
| -------- | ------------------ | -------------------------------------- |
| `select` | small sets         | oldest, fd-set limited                 |
| `poll`   | medium             | array of fds, no FD_SETSIZE limit      |
| `epoll`  | large (thousands)  | Linux-specific, most scalable          |

**Driver side** — implement the `.poll` file operation:

```c
static __poll_t my_poll(struct file *file, poll_table *wait)
{
    __poll_t mask = 0;
    poll_wait(file, &my_wq, wait);     // register this wait queue with the kernel
    if (data_ready)
        mask |= POLLIN | POLLRDNORM;   // tell user "readable"
    return mask;
}
```

**Writer / ISR side** — wake the same queue:

```c
data_ready = 1;
wake_up_interruptible(&my_wq);
```

---

## 4. Flow

```
User: poll(fds, n, timeout)
          │
          ▼
Kernel calls each driver's .poll  → poll_wait() registers wait queue
          │
          ▼ (no data yet → process sleeps)
Writer/ISR: data_ready = 1; wake_up_interruptible(&my_wq)
          │
          ▼
poll() returns → user sees POLLIN → calls read()
```

---

## 5. Interview Questions & Answers

**Q1. What is a wait queue (in the poll context)?**

> The kernel list that `poll_wait()` registers the process on. When the driver
> calls `wake_up_interruptible()` on that queue, the kernel re-evaluates `.poll`
> and unblocks the waiting `poll`/`select`/`epoll` call.

**Q2. What does `POLLIN` mean?**

> It is the event flag meaning "data is available to read" on this fd. The driver
> returns it from `.poll` when its buffer/condition has readable data;
> `POLLOUT` similarly means "writable".

**Q3. Difference between `poll`, `select`, and `epoll`?**

> `select` uses fixed-size fd sets (limited by `FD_SETSIZE`) and rescans all fds
> each call. `poll` uses an array of fds (no hard limit) but still scans linearly.
> `epoll` is Linux-specific and event-based — the kernel keeps the interest list,
> so it scales to thousands of fds with O(1) readiness notification.

**Q4. Why must the driver call `poll_wait()`?**

> `poll_wait()` adds the process to the driver's wait queue **without sleeping**.
> This lets the kernel know which queue to monitor so a later `wake_up` can
> re-check readiness. Without it, `poll` could block forever.

**Q5. "I used `wake_up_interruptible()` but only one process received data — why?**

> `wake_up_interruptible()` wakes **all** eligible waiters, but they all then
> compete for the **same shared data**. The first to run consumes it; the others
> wake, find nothing available, and go back to sleep. So user space may appear to
> show only one process getting the event even though several were woken
> internally. (This is the thundering-herd issue addressed in Program 9.)
